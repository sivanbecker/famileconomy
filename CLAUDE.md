# Claude Code Instructions — Famileconomy

## Project Overview

Famileconomy is a personal finance management app for families — tracking income, expenses, budgets,
savings goals, and shared accounts. It is a Turborepo monorepo with a Next.js web app, an Expo
mobile app, and a Node.js/Fastify API backed by PostgreSQL.

## Model Usage

- **Branching Workflow** and **Quality Gates** steps must use the **Haiku model** (`claude-haiku-4-5-20251001`).
- If any step reveals failing tests, lint errors, or formatting issues that require code changes,
  **switch to Sonnet** to diagnose and fix, then resume with Haiku.

## Branching Workflow

**Never work directly on `main`.** For every task or feature:

1. Create a dedicated branch from `main`:
   - `feat/<scope>-<description>` — new feature
   - `fix/<scope>-<description>` — bug fix
   - `perf/<scope>-<description>` — performance improvement
   - `docs/<description>` — documentation only
   - `chore/<description>` — maintenance
   - `mobile/<description>` — React Native–specific change
2. Do all work on that branch.
3. When complete, open a PR and merge back to `main`.

This rule applies to all sessions, agents, and subagents.

## Quality Gates (after every change)

After completing any change, always verify:

1. **Tests pass** — run `turbo test` and ensure all tests are green.
2. **Lint is clean** — run `turbo lint` with zero errors.
3. **Formatting is clean** — run `turbo format:check` and ensure no diffs.
4. **Types pass** — run `turbo typecheck` with zero errors.
5. **Migrations applied** — run `npm run db:migrate:status --workspace=api` and confirm no pending migrations. If any are pending, remind the user to run `npm run db:migrate --workspace=api`.

- Default to `gh pr merge <n> --merge --delete-branch` (true merge commit).
- Do **not** use `--squash` unless the user explicitly asks.
- `--rebase` is acceptable when the user wants linear history and the branch has a clean commit sequence.

## Commit Order

- Feature/fix changes and lint+format fixes **must be in separate commits**.
- Commit the feature/fix first, then commit lint/format-only changes: `style: lint and format fixes`.

## Coding Conventions

### TypeScript
- `strict: true` is mandatory. Never use `any`. Use `unknown` and narrow it.
- `exactOptionalPropertyTypes: true` is enabled — do not assign `undefined` to optional fields.
- `noUncheckedIndexedAccess: true` is enabled — always check array index access for undefined.
- Import types with `import type { ... }` to avoid importing runtime values as types.

### Naming
- Components: PascalCase (`TransactionCard`)
- Hooks: `use` prefix + camelCase (`useTransactions`, `useActiveAccount`)
- Utility functions: camelCase (`formatCurrency`, `toShekels`, `toShekels`)
- DB tables/columns: snake_case (enforced by PostgreSQL convention)
- Zod schemas: camelCase + `Schema` suffix (`transferSchema`, `loginSchema`)
- TypeScript types: PascalCase (`Transaction`, `Account`, `Budget`)

### File Structure (per app/package)
```
src/
  __tests__/      — unit tests (logic functions)
  components/
    layout/       — shared layout
  hooks/          — custom React hooks
  lib/            — pure utilities, API client, queryClient, schemas
  pages/ (web) or screens/ (mobile) — one folder per domain
  store/          — Zustand stores
  types/          — DB types + enriched types (use @famileconomy/types for shared types)
```

### Component Structure
- Keep local types, helper functions, and sub-components in the same file as the main component.
- Extract to shared packages only when used by more than one app.
- Use section comments `// ─── Section Name ────` to organize large files.

## Financial Math Rules

**CRITICAL: Never use floating-point arithmetic for money.**

- All monetary amounts are stored and processed as **integer shekels** (or the smallest currency unit).
- `amount: 1050` means 10.50 in the display currency. Never `amount: 10.50`.
- Use `toShekels(displayValue: number): number` and `fromShekels(shekels: number): string` utilities from `@famileconomy/utils`.
- Never do `price * quantity` with floats. Convert to shekels first, then multiply.
- Never store exchange rates as the multiplied result — store the rate separately and recalculate on read.
- Budget calculations must use `Math.floor` (never round up an amount you don't have).

## RTL / Hebrew UI Rules

- The app supports Hebrew (RTL) and English (LTR).
- `<html dir>` and `<html lang>` are set by `App.tsx` based on the active language — do not hardcode them.
- Use logical CSS properties (`margin-inline-start` not `margin-left`) in all custom CSS.
- Tailwind provides RTL utilities: `ltr:` and `rtl:` prefixes — use them for directional styles.
- In React Native: use `I18nManager.isRTL` to flip icon directions; do not hardcode left/right.
- All user-visible strings must go through i18next. Never hardcode display strings in components.
- Translation files: `src/locales/he/` and `src/locales/en/` — one JSON file per domain (common, transactions, budgets, accounts).

## Test Expectations

- Every pure function (calculations, formatters, validators) must have a unit test.
- Financial calculation functions require edge-case tests: zero, negative, maximum safe integer, rounding.
- Component tests use React Testing Library. Do not test implementation details — test behavior.
- The database client is mocked globally in `src/test/setup.ts`. Do not make real DB calls in unit tests.
- E2E tests (Playwright) cover critical paths: login, add transaction, view balance, create budget.
- Coverage threshold: 80% on `packages/utils` and `api/src/services` (pure logic). No threshold on UI.

## Security Rules

- **Never** interpolate user input into SQL strings. Always use parameterized queries (Prisma or `pg` bound params).
- **Never** store JWT tokens in localStorage or plain AsyncStorage. Use httpOnly cookies (web) or Expo SecureStore (mobile).
- **Never** log sensitive data (amounts, account numbers, passwords, tokens). Use `[REDACTED]` in structured logs. Log record IDs only.
- All Fastify route handlers must validate request bodies with Zod before touching the database.
- The `@fastify/helmet` plugin must be registered on every Fastify server instance.
- The `@fastify/rate-limit` plugin must be registered on all auth endpoints (`/auth/*`) and import endpoints.
- Password hashing must use **Argon2id** (not bcrypt, not SHA-256). Parameters: `memory: 65536`, `iterations: 3`, `parallelism: 4`.
- Refresh tokens are **one-time use** — rotate on every use, store hash in DB, revoke on logout and suspicious activity.
- Every write that touches financial data (transactions, recurring, income, budgets) must write an **audit log entry** — who changed what, when, and from what value to what value. Audit log records are append-only and never deleted.
- PostgreSQL Row Level Security (RLS) must be enabled as a second independent layer on all financial tables, even if application-layer scoping is also applied.
- All financial data columns in PostgreSQL must use pgcrypto encryption. Never add a financial column without confirming encryption requirement first.
- CORS must be explicitly configured — never use `origin: '*'` in production.
- Dependency security: `npm audit --audit-level=high` must pass in CI. Snyk `continue-on-error` must be `false` in production.

## Observability Rules

**Principle: instrument when the code is written, not after the first production incident.**

### Structured Logging
- All logs must be structured JSON (use `pino` — Fastify's built-in logger). Never `console.log` in production code.
- Every log entry must include: `requestId`, `userId` (if authenticated), `service`, `level`, `msg`, `durationMs` (on request completion).
- Never log financial amounts or PII. Log record IDs only.
- Log levels: `error` for unhandled exceptions, `warn` for recoverable errors (retry, validation fail), `info` for key business events (import completed, sync finished), `debug` for development only (disabled in production).

### Distributed Tracing
- Use **OpenTelemetry** SDK (`@opentelemetry/sdk-node`) for all tracing. Never vendor-lock to a specific backend.
- Instrument: HTTP requests (auto), database queries (Prisma plugin), BullMQ jobs (manual spans), external calls (R2 uploads).
- Propagate `traceparent` headers between services (API → workers).
- Free backend: **Grafana Tempo** (via Grafana Cloud free tier, 14-day retention, 50GB/month).
- Upgrade path: same OpenTelemetry SDK, swap exporter endpoint to Datadog / Honeycomb / Jaeger.

### Metrics
- Instrument with **OpenTelemetry Metrics** (counters, histograms). Key metrics to collect from day one:
  - `http_request_duration_ms` (histogram, by route + status)
  - `import_jobs_total` (counter, by status: success / duplicate / error)
  - `sync_duration_ms` (histogram)
  - `db_query_duration_ms` (histogram, by operation)
  - `auth_attempts_total` (counter, by result: success / fail / rate_limited)
- Free backend: **Grafana Cloud free tier** (10k series, 14-day retention).
- Upgrade path: same OpenTelemetry SDK, swap exporter to Datadog / Prometheus + Grafana self-hosted.

### Error Tracking
- Use **Sentry** free tier for exception capture and alerting. Initialize in API, web, and mobile apps.
- Set `sampleRate: 1.0` for errors, `tracesSampleRate: 0.1` in production (10% of traces to Sentry).
- Sentry is complementary to OpenTelemetry — use both. Sentry for exception detail + stack traces; OTel for spans + metrics.

### Alerting
- Free tier: **Better Stack** (Uptime monitoring + log-based alerts, free plan covers 10 monitors + 1 log destination).
- Alert on: API error rate > 1% over 5 min, import queue depth > 50 jobs, P99 response time > 2s, any 5xx on auth endpoints.
- Upgrade path: Grafana Cloud Alerting (same dashboards as metrics) or PagerDuty.

### Health Endpoints
- Every service must expose `GET /health` (liveness) and `GET /ready` (readiness — checks DB + Redis connectivity).
- Health endpoints must NOT require authentication.
- Railway and Vercel use these for zero-downtime deploys.

## Encryption Rules (Famileconomy-specific)

- Cloud DB: Financial data columns use pgcrypto encryption. Never add a
  financial column without confirming encryption requirements first.
- Mobile local DB: WatermelonDB uses SQLCipher. The encryption key is
  stored in Expo SecureStore — never in AsyncStorage.
- Never log decrypted financial values. Log record IDs only.

## TDD Workflow

**Every feature must follow strict test-driven development — no exceptions.**

1. **Write tests first.** Before any implementation, write unit tests that express the expected behavior. They will fail — that is intentional.
2. **Commit tests only.** Commit the failing tests and wait for the user to review and approve before writing any implementation code.
3. **Write implementation.** Only after the user approves the tests, write the code that makes them pass.
4. **Verify green.** Confirm all tests pass.
5. **Audit beyond green.** If the implementation passes the tests but is architecturally wrong — bad abstractions, unsafe patterns, violates distributed systems best practices (idempotency, consistency, retry safety, data integrity, security boundaries) — **flag it explicitly**. Do not silently accept a passing test suite. Point out what is wrong and suggest changes to both the tests and the implementation to align with best practices.

**If a requirement is unclear, flag it — do not guess.**

1. Stop before writing code.
2. State the assumption you would have made.
3. Ask the user to confirm or redirect.
