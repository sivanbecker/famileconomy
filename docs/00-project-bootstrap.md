## Section 1: CLAUDE.md — What to Carry Over

### Rule 1: Model Usage (Haiku for workflow / Sonnet for fixes)

> **Original rule:**
> "Branching Workflow and Quality Gates steps must use the Haiku model (`claude-haiku-4-5-20251001`). If any step reveals failing tests, lint errors, or formatting issues that require code changes, switch to the default model (Sonnet) to diagnose and fix, then resume with Haiku."

**Decision: Carry over as-is.**

This is an agentic cost-management rule. It applies universally regardless of tech stack: use Haiku for mechanical steps (git, running tests, checking lint), escalate to Sonnet only when code changes are needed. Saves significant API cost on long sessions.

---

### Rule 2: Branching Workflow — Never Work Directly on `main`

> **Original rule:**
> "Never work directly on `main`. For every task or feature: (1) Create a dedicated side branch from main. (2) Do all work on that branch. (3) When the task is complete, merge (or PR) back to main. This applies to all sessions, agents, and subagents."

**Decision: Carry over as-is.**

Branch protection is a universal good practice. The branch naming convention (`feat/`, `fix/`, `perf/`, `docs/`, `chore/`) used in this project has proven very clear and should be kept. Add `mobile/` as a prefix for React Native–specific branches.

---

### Rule 3: Quality Gates

> **Original rule:**
> "After completing any change, always verify: (1) Tests pass — run `npm test`. (2) Lint is clean — run `npm run lint`. (3) Formatting is clean — run `npm run format`. (4) Migrations applied — run `supabase migration list`."

**Decision: Adapt for Turborepo.**

The gate logic is correct; the commands need adaptation:

- Replace `npm test` with `turbo test` (or per-package: `turbo test --filter=<app>`)
- Replace `npm run lint` with `turbo lint`
- Replace `npm run format` with `turbo format`
- Replace `supabase migration list` with the appropriate Fastify/pg-migrate equivalent (e.g., `npm run db:migrate:status` in the `api` package)

---

### Rule 4: PR Description — No Claude Attribution

> **Original rule:**
> "Never include lines like 'Generated by Claude Code', '🤖 Generated with ...', 'Co-Authored-By: Claude...' or any similar attribution in PR descriptions."

**Decision: Carry over as-is.**

Personal preference of the project owner. Non-negotiable.

---

### Rule 5: Merge Strategy

> **Original rule:**
> "Default to `gh pr merge <n> --merge --delete-branch` (true merge commit). Do not use `--squash` unless the user explicitly asks. `--rebase` acceptable for linear history."

**Decision: Carry over as-is.**

This rule exists specifically because `--squash` makes history look like work happened directly on `main`, obscuring which PR a commit came from. This is even more important in a monorepo where multiple teams/apps contribute to the same `main`.

---

### Rule 6: Commit Order (feature first, style second)

> **Original rule:**
> "Feature/fix changes and lint+format fixes must be in separate commits. Commit the feature/fix first, then commit lint/format-only changes in a follow-up commit with a message like `style: lint and format fixes`."

**Decision: Carry over as-is.**

Keeps blame clean. Essential when bisecting bugs in a monorepo.

### Additional Security Tooling — Recommended for Famileconomy (Not in Current Project)

#### 0. Argon2id for password hashing (not bcrypt)

**What it adds:** Argon2id is the winner of the Password Hashing Competition and is memory-hard, making GPU and ASIC attacks impractical. Bcrypt is fine for a basic app but is not the current best practice for a financial application.

Parameters: `memory: 65536` (64 MB), `iterations: 3`, `parallelism: 4`.

```bash
npm install argon2 --workspace=api
```

Never use `crypto.createHash('sha256')` or `bcrypt` for passwords in this project. This applies to all sessions and subagents — do not implement bcrypt even as a placeholder.

---

#### 0b. One-time-use refresh token rotation

**What it adds:** Refresh tokens are stored as a hash in the DB alongside the `deviceId` and a `revokedAt` field. On every token refresh:

1. Look up the token hash in DB — if not found or `revokedAt` is set → revoke entire session.
2. Issue new access token + new refresh token.
3. Invalidate the old refresh token hash in DB immediately.
4. This makes stolen refresh tokens unusable after one use (the legitimate client will have already rotated it).

This is a **must-have** before any public beta — stolen session tokens are the primary attack vector on financial apps.

---

#### 0c. Append-only audit log table

**What it adds:** Every mutation of financial data (create/update/delete on `transactions`, `recurring_expenses`, `income`, `budgets`) writes a row to `audit_log`:

```sql
CREATE TABLE audit_log (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid NOT NULL REFERENCES users(id),
  table_name  text NOT NULL,
  record_id   uuid NOT NULL,
  action      text NOT NULL CHECK (action IN ('INSERT', 'UPDATE', 'DELETE')),
  old_value   jsonb,          -- encrypted, NULL for inserts
  new_value   jsonb,          -- encrypted, NULL for deletes
  performed_by uuid NOT NULL REFERENCES users(id),
  performed_at timestamptz NOT NULL DEFAULT now(),
  request_id  text            -- OpenTelemetry trace ID for correlation
);
-- No UPDATE or DELETE permissions granted on audit_log to application role
```

Audit logs must never be deleted by application code (only by an explicit retention policy run by a privileged admin role). This satisfies Israeli Privacy Protection Law's requirement for data traceability and is essential for debugging financial discrepancies.

## Section 5b: Observability Tooling — What to Add from Day One

Observability is not a phase-2 concern for a financial app. A missing transaction, a silent import failure, or a slow sync are bugs that users will notice before you will — unless you have visibility into the system from the start.

The strategy is: **OpenTelemetry everywhere, free backends now, paid backends on demand.**

OpenTelemetry (OTel) is the CNCF-standard vendor-neutral observability SDK. By instrumenting with OTel from day one, you can switch backends (Datadog, Honeycomb, Jaeger, Grafana) by changing one environment variable — no code changes.

---

### The Three Pillars

#### Pillar 1: Structured Logging

**Tool: `pino`** (already Fastify's default logger — zero extra install)

`pino` produces newline-delimited JSON logs at ~10x the throughput of `winston`. Every log line is a structured JSON object, queryable by any log aggregator.

**Free backend: Better Stack (Logtail)** — free plan includes 1 GB/day ingestion, 3-day live tail, and log-based alerts. Upgrade to paid for longer retention.

```bash
npm install pino-loki --workspace=api   # Loki transport (for Grafana Cloud)
# OR
# Use Better Stack's pino transport: @logtail/pino
npm install @logtail/pino --workspace=api
```

**Log destinations:**

- Development: `pino-pretty` (human-readable stdout)
- Staging/Production: JSON stdout → collected by Railway/Vercel log drains → shipped to Better Stack

**What to log (with `info` level):**

- Request completed: `{ requestId, userId, method, route, statusCode, durationMs }`
- Import finished: `{ requestId, userId, batchId, imported, duplicates, matched, durationMs }`
- Sync completed: `{ requestId, userId, pushed, pulled, conflicts, durationMs }`
- Recurring matched: `{ recurringId, transactionId, matchScore }`
- Auth event: `{ userId, event: 'login_success' | 'login_fail' | 'token_rotated' | 'session_revoked' }`

**What to log at `warn` level:**

- Validation error on API boundary (Zod rejection)
- Retry triggered (DB connection retry, BullMQ job retry)
- Rate limit hit: `{ ip, endpoint, limit }`

**What to log at `error` level:**

- Unhandled exception (include `err.stack` but never raw financial data)
- Job failed after max retries: `{ jobId, queue, attempt, err.message }`

#### Pillar 2: Distributed Tracing

**Tool: OpenTelemetry Node.js SDK**

```bash
npm install @opentelemetry/sdk-node \
            @opentelemetry/auto-instrumentations-node \
            @opentelemetry/exporter-trace-otlp-http \
            --workspace=api
```

Auto-instrumentation covers: HTTP (Fastify), PostgreSQL (`pg`), Redis (`ioredis`), and most npm libraries with zero code changes. Manual spans are added for business-logic boundaries.

**Free backend: Grafana Cloud free tier** — 50 GB traces/month, 14-day retention. Sign up at grafana.com (no credit card for free tier).

**What to add manual spans for:**

- Each BullMQ job: `csv-import`, `matching`, `forecast`, `notifications`
- Each parser's `parse()` call: `max-parser.parse`, `cal-parser.parse`
- The deduplication loop: `dedup.checkHash`
- The matching engine: `matcher.match`
- Each outbound call to R2 (upload, download, delete)

**Span attributes to include (sanitized — no financial values):**

## Section 6: Versioning & Release Strategy

### Current Versioning Approach

The current project uses **release-please** with the `node` release type. Every `push` to `main` triggers the `release-please` GitHub Action, which:

1. Reads all conventional commits since the last release tag
2. Determines the version bump (MAJOR/MINOR/PATCH) from commit types
3. Opens a "Release PR" with an updated `CHANGELOG.md` and bumped `package.json`
4. When the Release PR is merged, it creates a git tag (e.g., `v0.10.0`) and a GitHub release

**Tags in the repo:** `v0.10.0`, `shopping-recipes-events-v0.9.2`, `shopping-recipes-events-v0.9.1`

**Interesting note:** The `.release-please-manifest.json` still shows `"0.9.2"` while `package.json` is `0.10.0` — there is a sync drift. This happened because the manual `chore: release v0.10.0` commit bypassed release-please. The manifest should be kept in sync with the actual released version.

---

### Recommended Strategy for Famileconomy

**Tool: release-please** (keep the same tool — it works well)

**Monorepo versioning approach: Unified version** — one version number for the whole Famileconomy product (not per-package). Rationale: Famileconomy is a product, not a library. Users don't install individual packages; they use the app. A single version like `v1.2.0` is meaningful to users and simpler to communicate.

**Conventional commit types for CHANGELOG:**

- `feat:` → MINOR bump, appears in changelog under "Features"
- `fix:` → PATCH bump, appears under "Bug Fixes"
- `perf:` → PATCH bump, appears under "Performance"
- `refactor:` → PATCH bump, appears under "Refactors"
- `docs:` → PATCH bump, appears under "Documentation" (not hidden — architecture decisions matter in a finance app)
- `chore:`, `style:`, `test:`, `ci:`, `build:` → PATCH bump, hidden in changelog

**Breaking changes:** Use `feat!:` or `fix!:` with `BREAKING CHANGE:` in the commit body to trigger MAJOR bump.

**Config files:**

```json

  "changelog-sections": [
    { "type": "feat", "section": "Features" },
```

### TypeScript Type Patterns

**Pattern: DB types are generated; enriched types are composed.**

`database.ts` contains raw table shapes (auto-generated or manually maintained). `index.ts` re-exports them as convenience aliases and defines "enriched" types (joins) used by the UI:

```ts
// types/index.ts pattern
export type Transaction = Database['public']['Tables']['transactions']['Row']

// Enriched for UI
export type TransactionWithCategory = Transaction & {
  category: Category | null
  account: Account
}
```

Never define types inline in components. Always import from `@/types` (or the shared `@famileconomy/types` package).

---

### Zod Schema Co-location

Form validation schemas live in `src/lib/schemas.ts` alongside their inferred TypeScript types:

```ts
export const transferSchema = z.object({
  amount: z.number().int().positive(), // always cents, never floats
  from_account_id: z.string().uuid(),
  to_account_id: z.string().uuid(),
  note: z.string().max(500).optional(),
})
export type TransferData = z.infer<typeof transferSchema>
```

In Famileconomy, schemas live in the shared `@famileconomy/types` package so frontend and API validate against the same schema.

---

### React Query Usage Pattern

- `queryClient.ts` configures a single shared `QueryClient` with sane defaults
- `staleTime: 2 minutes`, `gcTime: 10 minutes`, `retry: 1`, `refetchOnWindowFocus: true`
- Query keys are arrays: `['transactions', accountId]`, `['accounts', userId]`
- Mutations use `onSuccess` to invalidate affected query keys, never manually refetch

```ts
// Pattern used in current project
const { data: lists } = useQuery({
  queryKey: ['shopping_lists', user.id],
  queryFn: () => supabase.from('shopping_lists').select('...').eq('owner_id', user.id),
})
```

### Component Structure Pattern (from `ListsPage.tsx`)

Files are organized with clear section comments when a file contains multiple concerns:

```ts
// ─── Types ────────────────────────
// ─── Helpers ──────────────────────
// ─── Sub-component ────────────────
// ─── Main component ───────────────
```

Helper functions and local types are defined in the same file as the component that uses them, not extracted until they're shared by >1 component.

### Auth Pattern: Context + Provider + Hook

```ts
// AuthContext.ts — interface definition only
// AuthProvider.tsx — state logic + Supabase subscription
// useAuth.ts — throws if used outside provider
export function useAuth() {
  const ctx = useContext(AuthContext)
  if (!ctx) throw new Error('useAuth must be used inside <AuthProvider>')
  return ctx
}
```

Carry this pattern exactly. Apply to any other global context (e.g., `useCurrentUser`, `useActiveAccount`).

### Supabase/DB Client Pattern

The Supabase client is initialized once in `src/lib/supabase.ts` and exported as a singleton. It handles the "not configured" case gracefully (logs a warning instead of crashing) so the UI is still visible during initial setup.

**For Famileconomy:** Apply the same grace-degradation pattern to the Fastify API: if `DATABASE_URL` is missing at startup, log a clear warning and refuse health checks rather than crashing silently.

---

### RTL/i18n Pattern

`App.tsx` syncs `<html dir>` and `<html lang>` with the active language on every language change:

```ts
useEffect(() => {
  const isHebrew = lang === 'he' || lang.startsWith('he-')
  document.documentElement.lang = isHebrew ? 'he' : 'en'
  document.documentElement.dir = isHebrew ? 'rtl' : 'ltr'
}, [i18n.language])
```

**For React Native (Expo):** `I18nManager.forceRTL(true)` on app launch when Hebrew is active. RTL layout is native in React Native; you do not use `dir="rtl"`.

---

### Lazy Loading Pattern

All pages except the first-render page are lazy-loaded:

```ts
// Eager — first paint
import AuthPage from './pages/Auth/AuthPage'

// Lazy — all others
const TransactionsPage = lazy(() => import('./pages/Transactions/TransactionsPage'))
```

Carry this to Next.js via `dynamic(() => import(...), { ssr: false })` for client-heavy pages. In React Native, React.lazy is not supported; use Expo Router's built-in lazy loading.

---

### Test Pattern

Unit tests are in `src/__tests__/` and test **logic functions extracted from components**, not the components themselves directly. The Supabase client is mocked globally in `src/test/setup.ts`.

```ts
// Extract pure function from component, test it in isolation
function progressPct(done: number, total: number): number {
  return total === 0 ? 0 : Math.round((done / total) * 100)
}

describe('progressPct', () => { ... })
```

This pattern is very clean and should be the standard for Famileconomy. Business logic functions (calculate interest, format currency, apply exchange rate) must be extracted and unit-tested independently of any UI.

## Section 8: What NOT to Carry Over

### 1. Supabase as the backend

**Leave behind. Do not use for Famileconomy.**

Supabase (BaaS) was appropriate for a small personal project with no server-side business logic. Famileconomy requires:

- Custom financial calculation logic (budgets, projections, recurring transactions, FX rates)
- Strict audit logging (every penny must be traceable)
- Multi-tenant row isolation that must be enforced at the application layer, not just RLS policies
- The ability to run complex SQL queries and transactions that span multiple tables atomically

A dedicated Node.js/Fastify API with direct PostgreSQL access is the correct architecture. Supabase Realtime is convenient but adds a dependency; use server-sent events or WebSockets from Fastify instead.

---

### 2. The manually-maintained `src/types/database.ts` pattern

**Adapt this.**

The current `database.ts` is a manually written placeholder with a comment saying "replace with `npx supabase gen types`". This is fine for a small hobby project but becomes a maintenance burden as the schema grows.

For Famileconomy, generate types automatically from the database on every migration:

```bash
# in api/package.json scripts:
"db:generate-types": "pg-to-ts generate --connection $DATABASE_URL --output packages/types/src/database.ts"
```

Use `pg-to-ts` or `Kysely`'s introspection to keep types in sync. Never manually maintain DB type files.

---

### 3. `pre-commit` hook running the full test suite

**Leave behind.**

The current `.husky/pre-commit` runs `npm test` (the full Vitest suite). This is slow. Developers will add `--no-verify` to every commit within a week. Pre-commit should run only `lint-staged`. Full tests belong in CI and optionally `pre-push`.

---

### 4. Snyk with `continue-on-error: true`

**Fix before shipping.**

A security scanner that doesn't block on failures provides false confidence. For a financial app, `continue-on-error: false` on the Snyk job is non-negotiable in production environments.

---

### 5. Vite as the bundler

**Not applicable to the new stack.**

Famileconomy uses Next.js (built on Webpack/Turbopack) for the web app and Expo (Metro) for mobile. Vite is not used. Do not bring over `vite.config.ts`.

---

### 6. Client-side Supabase Auth (`supabase.auth.signIn`)

**Not applicable.**

The current app handles auth entirely on the client using the Supabase JS client. Famileconomy with a custom Fastify backend needs server-side auth: JWT + refresh tokens issued by Fastify, stored in httpOnly cookies (not localStorage). Client-side token storage is a security antipattern for financial apps.

---

### 7. The `.release-please-manifest.json` drift issue

**Do not replicate.**

The current project has a sync drift: `package.json` says `0.10.0` but `.release-please-manifest.json` says `0.9.2`. This happens when you manually bump the version without going through release-please. In Famileconomy, establish a rule: **never manually bump the version in `package.json`**. Let release-please own it entirely.

---

### 8. Floating shopping-specific UI assumptions

**Context-specific, do not copy.**

The RTL support, Hebrew locale strings in `src/locales/he/`, the `font-family: Rubik` choice, and the Hebrew category management system are specific to a Hebrew-language shopping app. Famileconomy may need i18n (Hebrew + English) but the specific translation keys, category names, and voice input for Hebrew product names do not apply.

---

### 9. `docs/OWASP_SCAN_2026-04-19.md` manual scan

**Replace with automated tooling.**

The current project ran a one-time manual OWASP analysis stored as a doc. For Famileconomy, automate this with OWASP ZAP on a weekly schedule against staging. Manual static analysis docs go stale immediately.

```markdown
   - `fix/<scope>-<description>` — bug fix
   - `perf/<scope>-<description>` — performance improvement
   - `docs/<description>` — documentation only
   - `chore/<description>` — maintenance
   - `mobile/<description>` — React Native–specific change
2. Do all work on that branch.
3. When complete, open a PR and merge back to `main`.

This rule applies to all sessions, agents, and subagents.

## Quality Gates (after every change)

After completing any change, always verify:

1. **Tests pass** — run `turbo test` and ensure all tests are green.
2. **Lint is clean** — run `turbo lint` with zero errors.
3. **Formatting is clean** — run `turbo format:check` and ensure no diffs.
4. **Types pass** — run `turbo typecheck` with zero errors.
5. **Migrations applied** — run `npm run db:migrate:status --workspace=api` and confirm no pending migrations. If any are pending, remind the user to run `npm run db:migrate --workspace=api`.

- `--rebase` is acceptable when the user wants linear history and the branch has a clean commit sequence.

## Commit Order

- Feature/fix changes and lint+format fixes **must be in separate commits**.
- Commit the feature/fix first, then commit lint/format-only changes: `style: lint and format fixes`.

## Coding Conventions

### TypeScript

- `strict: true` is mandatory. Never use `any`. Use `unknown` and narrow it.
- `exactOptionalPropertyTypes: true` is enabled — do not assign `undefined` to optional fields.
- `noUncheckedIndexedAccess: true` is enabled — always check array index access for undefined.
- Import types with `import type { ... }` to avoid importing runtime values as types.

### Naming

- Components: PascalCase (`TransactionCard`)
- Hooks: `use` prefix + camelCase (`useTransactions`, `useActiveAccount`)
- Utility functions: camelCase (`formatCurrency`, `toCents`)
- DB tables/columns: snake_case (enforced by PostgreSQL convention)
- Zod schemas: camelCase + `Schema` suffix (`transferSchema`, `loginSchema`)
- TypeScript types: PascalCase (`Transaction`, `Account`, `Budget`)

### File Structure (per app/package)
```

src/
**tests**/ — unit tests (logic functions)
components/
layout/ — shared layout
hooks/ — custom React hooks
lib/ — pure utilities, API client, queryClient, schemas
pages/ (web) or screens/ (mobile) — one folder per domain
store/ — Zustand stores
types/ — DB types + enriched types (use @famileconomy/types for shared types)

```

### Component Structure
- Keep local types, helper functions, and sub-components in the same file as the main component.
- Extract to shared packages only when used by more than one app.
- Use section comments `// ─── Section Name ────` to organize large files.

## Financial Math Rules

**CRITICAL: Never use floating-point arithmetic for money.**

- All monetary amounts are stored and processed as **integer cents** (or the smallest currency unit).
- `amount: 1050` means 10.50 in the display currency. Never `amount: 10.50`.
- Use `toCents(displayValue: number): number` and `fromCents(cents: number): string` utilities from `@famileconomy/utils`.
- Never do `price * quantity` with floats. Convert to cents first, then multiply.
- Never store exchange rates as the multiplied result — store the rate separately and recalculate on read.
- Budget calculations must use `Math.floor` (never round up an amount you don't have).

## RTL / Hebrew UI Rules

- The app supports Hebrew (RTL) and English (LTR).
- `<html dir>` and `<html lang>` are set by `App.tsx` based on the active language — do not hardcode them.
- Use logical CSS properties (`margin-inline-start` not `margin-left`) in all custom CSS.
- Tailwind provides RTL utilities: `ltr:` and `rtl:` prefixes — use them for directional styles.
- In React Native: use `I18nManager.isRTL` to flip icon directions; do not hardcode left/right.
- All user-visible strings must go through i18next. Never hardcode display strings in components.
- Translation files: `src/locales/he/` and `src/locales/en/` — one JSON file per domain (common, transactions, budgets, accounts).

## Test Expectations

- Every pure function (calculations, formatters, validators) must have a unit test.
- Financial calculation functions require edge-case tests: zero, negative, maximum safe integer, rounding.
- Component tests use React Testing Library. Do not test implementation details — test behavior.
- The database client is mocked globally in `src/test/setup.ts`. Do not make real DB calls in unit tests.
- E2E tests (Playwright) cover critical paths: login, add transaction, view balance, create budget.
- Coverage threshold: 80% on `packages/utils` and `api/src/services` (pure logic). No threshold on UI.

## Security Rules

- **Never** interpolate user input into SQL strings. Always use parameterized queries.
- **Never** store JWT tokens in localStorage. Use httpOnly cookies only.
- **Never** log sensitive data (amounts, account numbers, passwords). Use `[REDACTED]` in logs.
- All Fastify route handlers must validate request bodies with Zod before touching the database.
- The `@fastify/helmet` plugin must be registered on every Fastify server.
- The `@fastify/rate-limit` plugin must be registered on all auth endpoints.
```
